1 · Executive summary

iPayHub (PaymentHub) is a multi-tenant, workflow-driven payment hub built for UAE commercial banks. Its defining idea is that a payment is described as data, not code: each payment flow is a JSON DSL workflow that a single generic Temporal workflow interprets at runtime, dispatching each step as a Temporal activity to a platform- or bank-specific worker.

This DSL-driven model means new payment products and new tenant banks are onboarded primarily through configuration, workflow definitions and pluggable adaptors rather than core code changes. The platform serves UAE schemes — IPI (Aani), FTS, WPS, NPSS, eDDA, CBFT/XBDR — across tenant banks ADIB, DIB, ENBD and FAB.

Architecturally significant characteristics

  • One generic workflow interpreter (PaymentDslWorkflow) executes every scheme — no per-scheme engine. verified
  • Clean Architecture layering (API → Application → Domain → Infrastructure) across all services. verified
  • Adaptor Interface Pattern isolates the core from bank CBS/AML/FX/RTGS/SFTP systems (ADR-003). verified
  • Append-only / CQRS projection data model — state changes are new rows; no DB foreign keys. verified
  • Saga compensation for rollback, driven by the DSL and a Temporal compensation stack. verified
  • Multi-tenant schema isolation in a single SQL Server database (ADR-004). verified
  • Fail-closed, hash-chained, tamper-evident audit (SQL Server 2022 Ledger + external witnesses). verified hardening partial
  • Defence-in-depth security: YARP gateway, RS256 JWT, RBAC, rate limiting, PII redaction, Vault secrets. verified
  • Two-collector OpenTelemetry topology into SigNoz/ClickHouse, decoupling services from the backend. verified

The stack is .NET 10 (C# 14) · Temporal · SQL Server 2022 · PostgreSQL (Temporal) · YARP · React · Redis · MQTT/Mosquitto · HashiCorp Vault · OpenTelemetry + SigNoz/Jaeger, orchestrated via layered Docker Compose and targeting Kubernetes for higher tiers.

2 · Architecture principles & tenets

These principles are load-bearing and enforced in code review, CI and the data model (from .github/copilot-instructions.md, the ADRs and the design corpus).

2.1 Data-integrity principles (hard constraints)

#PrincipleRationale
P1Append-only operational data. State/status/balance changes are new timeline/event rows — never in-place UPDATE on transaction tables.Eliminates lock contention at scale; complete reconstructable history.
P2Separate immutable facts from mutable state. Status fields never share a table with immutable transaction data.Enables CQRS projection and tamper-evidence.
P3No database-level foreign keys. Referential integrity is enforced at the service layer via soft references.Avoids cascading deadlocks and cross-rail coupling.
P4Designed for millions of rows. BIGINT IDENTITY PKs, bounded NVARCHAR, DECIMAL(18,4) money, nvarchar(max) only for JSON/binary.Throughput & predictable indexing.
P5Phase-gate refactors. Complete phase N and its deliverable before phase N+1.Controlled, reviewable change.

2.2 Design tenets

  • Configuration over code — products/tenants added via DSL, the 4-level config hierarchy and adaptor packages.
  • One engine, many workflows — a single Temporal workflow interprets all DSL definitions.
  • Pluggability at the edges — every external system is reached through a typed adaptor resolved at runtime.
  • Determinism & immutability — workflow changes use Patched()/GetVersion(); published versions are immutable.
  • Fail-closed for audit and money-movement; AI autonomy is deny-by-default.
  • Defence-in-depth & least privilege — client-side checks are never the security boundary.
  • Observability first-class but decoupled — no hard runtime dependency on the monitoring backend.
  • Portability — Docker/Kubernetes, no hard cloud lock-in; one image across tiers.

4 · High-level system architecture

PaymentHub is a layered, tiered system. Clients reach a single YARP gateway; the gateway fronts Clean-Architecture microservices; payment execution is delegated to Temporal workers that interpret the DSL; everything persists to a multi-schema SQL Server (with PostgreSQL backing Temporal) and emits telemetry through a two-collector OpenTelemetry pipeline.

7 · Key architectural patterns

7.1 DSL-driven workflow orchestration

A workflow is JSON in dsl/workflows/*.json (schema payment-dsl-v2.schema.json): workflow_name, payment_scheme, metadata.bank_id, variables, and an ordered steps[] array supporting sequential, conditional, parallel and fan-out steps with input_mapping, output_mapping, retry_policy, compensation_activity and condition gates. A single generic PaymentDslWorkflow interprets it for all schemes; the pure interpreter logic is factored into DslInterpreter.cs for testability.

7.2 Clean Architecture

.API            Minimal API endpoints, DTOs — thin, no business logic
.Application    Use cases, services, interfaces, workflow/interpreter logic, CQRS
.Domain         Entities, value objects — zero external dependencies
.Infrastructure EF Core DbContext, repositories, adaptors, external clients

7.3 Adaptor Interface Pattern (ADR-003)

Each external category is a typed interface — ICoreBankingAdaptor, IAmlScreeningAdaptor, IFxRateAdaptor, IRtgsAdaptor, ISftpAdaptor, INotificationAdaptor, ISwiftAdaptor. The core depends only on interfaces; banks deliver a versioned package (extension DLL, adaptor DLL, config bundle, manifest) supporting multiple entities (ISB/CONV). Each call is wrapped with Polly (retry, timeout, circuit-breaker) and OTel tracing.

7.4 CQRS & append-only projection

Workflows emit ProjectTransaction / ProjectTransactionDetail to the Operations API, which UPSERTs a platform.Transactions header and appends to platform.TransactionTimeline (one row per state change), plus insert-once per-scheme rail tables. High-throughput inserts are batched via Channel<T> TVP flushers.

7.5 Saga compensation

Each step may declare a compensation_activity. On failure at step N, the orchestrator unwinds completed steps LIFO. Complex reversals use dedicated sub-workflows (adib-ipi-auto-reverse, adib-ipi-nak-reversal, wps-posting-retry).

7.6 Multi-tenancy & the 5-layer framework (ADR-006)

Layer 5  Environment configuration   (conn strings, flags, secrets) — highest priority
Layer 4  Bank extension package      (custom adaptors, workflow overrides, plugins)
Layer 3  Bank configuration          (cut-off times, limits, nostro, XSLT maps)
Layer 2  Product definitions         (DSL, validation, message schemas per scheme)
Layer 1  Core engine (locked)        (services, Temporal, YARP, adaptor runtime)

Resolution is Environment > Bank > Product > Platform, implemented by the Configuration service with maker-checker approval and MQTT hot reload. New banks onboard by configuration + schema provisioning + worker deployment — no core code changes.

7.7 Fail-closed, tamper-evident audit

Audit is on-critical-path: a failed critical write blocks the operation. State is append-only, hash-chained (SHA-256) and anchored to external witnesses (see §9.4).

7.8 Bounded-autonomy AI

The AI Workflow Builder runs a ReAct agent with deny-by-default boundaries: bounded iterations/session, max steps/workflow, scheme allowlist, minimum guardrail pass score (~0.7), and escalate (fail-closed) on uncertainty. AI decisions are recorded to a dedicated AiDecisionAudit ledger.

8 · Workflow & orchestration architecture

8.1 Execution flow — IPI outward (representative)

8.2 Deterministic versioning

Behavioural changes to PaymentDslWorkflow use Workflow.Patched() / Workflow.GetVersion(changeId, min, max), guarded by if (_effectiveVersion >= N). Existing markers are never modified or removed. CI enforces invariants (TemporalVersioningInvariantTests): unique change-IDs, shipped change-IDs present, CurrentCodeVersion matches the highest maxSupported, no DEFAULT_VERSION (-1) as minSupported.

8.3 Workflow governance (4-eyes)

Draft → Review → Approved → Production → Deprecated with separation of duty (reviewer ≠ submitter; promoter ≠ reviewer). Production workflows are immutable; updates throw WorkflowImmutabilityException and require a new version.

8.4 Signals & queries

Signals: Pause, Resume, Stop, CancelWorkflow (triggers compensation), RestartFromStep (idempotency pre-check), InjectState. Query: GetState returns a snapshot.

9.2 Multi-schema design (ADR-004)

  • Tenant schemas: adib, dib, enbd, fab — per-bank transaction/posting data, fully isolated.
  • Shared platform schema: platform — IAM, config, workflow definitions, audit, participants, reporting, projection headers/timeline (tenant-filtered).
  • Concern schemas: exception and per-scheme rail schemas (ipi, fts, wps, …) for insert-once rail tables.
The database-architecture.md design describes a broader target (~22 schemas / 200+ tables with monthly partitioning, RCSI, computed-column indexing). That is the design intent / scaling blueprint; the authoritative live schema is whatever the latest db-script/ baseline provisions. This document presents the principles and verified core schemas and cites the broader counts as targets, not asserted fact.

9.3 Append-only / CQRS projection model

No DB foreign keys; integrity enforced in the service layer via business-ID soft references; idempotent writes use NOT EXISTS dedup keyed on (TransactionId, StepId, NewStatus) so Temporal retries are safe.

9.4 Tamper-evident audit ledger

A dual-channel model separates legal audit from operational telemetry, linked only by CorrelationId (= Activity.Current.TraceId):

  • Fail-closed ingestion — critical writes retry 3× then raise AuditWriteException and block the operation.
  • Hash chain + witnesses — SHA-256 + Merkle batching; adapters: NoOp (dev), OpenTimestamps (default prod), RFC 3161 TSA, WORM Blob, external webhook, Azure Confidential Ledger.
  • SQL Server 2022 Ledger tables — AuditLog_Ledger, AiDecisionAudit_Ledger, TransactionJournal_Ledger, TokenVaultAccess_Ledger with sp_verify_database_ledger.
  • PII tokenisation — TokenVault (AES-256) supports GDPR Art.17 erasure without breaking the chain.
  • Transport security — monotonic SequenceId + HMAC-SHA256 envelope with constant-time verification.
  • Tiered retention — audit/AI/journal long-term (7 yr in requirements, 10 yr/CBUAE in audit design), raw payloads ~1 yr, OTel spans ~90 d.

10 · Integration architecture (Smart Integration / Nexus)

Core platform code must never change when a bank adds, removes or replaces an external-system integration.

IntegrationProfile (ADR-009) is the first-class entity representing a connection: type (CBS/FTS/UAEFTS/Aani/WPS/Participant), connection config (endpoint, protocol, auth-credential reference), transformation rules, schema version, health status and tenant — a single canonical inventory enabling drift detection and ops-console health views.

10.1 Layered transformation pipeline (target design)

Inbound channels → Channel listeners → Raw message store (immutable, encrypted)
   → Message converter → CINJ (Canonical Internal JSON)
   → Temporal workflow → Smart Integration (sole outbound gateway)
   → Message formatter → Adaptor registry → Third-party systems

Smart Integration is the sole outbound gateway, applying idempotency (Redis), PII masking (Vault Transit tokenise / HMAC pseudonymise), per-tenant bulkheads, circuit-breaking, a DLQ, and audit for every operation.

10.2 Async activity-completion pattern

An activity grabs the Temporal task token, persists it to async_pending_calls, publishes the outbound request and suspends via CompleteAsyncException. When the external system replies, a reply-channel listener looks up the token and calls CompleteActivityAsync — keeping workers free during multi-minute waits while preserving exactly-once completion.

12 · Observability architecture

12.1 Two-collector OTel topology (ADR-008)

The product-side collector has a locked processor list (memory_limiter → resource → batch) performing only resource-attribute stamping — no data transformation; changes beyond batch tuning require an ADR amendment. Services therefore have no hard runtime dependency on SigNoz, and the topology satisfies the CBUAE requirement to isolate payment-system telemetry from infrastructure monitoring.

12.2 Instrumentation standards

  • One ActivitySource and one Meter per service, named PaymentHub.<Service>.
  • Span attributes dot-separated lowercase (payment.scheme); no raw PII.
  • Metrics paymenthub.<domain>.<name>, always dimensioned.
  • Structured logging via ILogger<T>, named parameters only, PII-redaction processors, standard tags + StepId/TxId/Scheme.

12.3 Monitoring & alerting

Gateway /health aggregation; thresholds for API latency (P95 > 500 ms), Temporal queue depth (> 1000), DB pool (> 80%), disk (> 85%), 5xx rate (> 1%), circuit-breaker OPEN (P1).

13.4 Deployment & tenancy models

The same images support three tenancy models with no code change: single-tenant (one platform instance per bank), multi-tenant (shared instance — schema-per-bank + per-bank workers), and hybrid (shared platform with dedicated workers per bank). Onboarding a bank is configuration + schema provisioning + worker deployment.

  • Tenant isolation — schema-per-bank for data, worker-per-bank for compute; workers may be shared in cost-sensitive deployments.
  • Data sovereignty — schema-per-bank enables per-bank backup, archival and geographic placement policies.
  • Independent upgrades — DSL workflows are versioned, so a bank can run an older workflow version while the platform core is upgraded.
  • Per-tenant observability — all traces / logs / metrics carry bank identifiers; per-bank dashboards and alerts without per-bank infrastructure.

14.4 Maintainability & portability

Zero-downtime deployment (rolling updates + Temporal worker versioning); on-prem and cloud portability; builds enforce TreatWarningsAsErrors, Nullable, AnalysisLevel=latest-recommended, EnforceCodeStyleInBuild.

14.5 Cost & licensing

The stack is predominantly open-source — Temporal (MIT), PostgreSQL, Redis (BSD), Mosquitto (EPL), YARP (MIT), SigNoz / ClickHouse (Apache-2.0) — with SQL Server the only commercially-licensed component (chosen for regulatory familiarity and Ledger tables; Standard Edition suffices for most deployments). OpenTimestamps is the free default audit witness (premium witnesses optional). The multi-LLM abstraction lets clients run local models (Ollama) to avoid API cost or route to cloud providers per environment. Containerised deployment avoids PaaS lock-in (bare metal, VMs or any Kubernetes); schema-per-bank avoids per-bank database-instance cost while preserving isolation.
